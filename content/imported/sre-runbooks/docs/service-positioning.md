# Service Positioning

## One-line Positioning

Mizumoto Tech provides cross-border cloud reliability engineering services for
growing teams, using senior Japan-side engineering judgment, low-sensitivity
delivery support, and auditable handover assets.

## Service Scope (v0.2)

### Entry Reviews

1. Cloud Cost Hygiene Pack
2. Architecture Review Pack

### Implementation Packs

1. Terraform Foundation Pack
2. CI/CD Bootstrap Pack

### Operational Readiness Pack

1. Day-2 Operations Handover Pack

### Future Retainer

1. Monthly Technical Advisory

## Phase-1 Delivery Boundary

- Delivery promise is GCP-first in Phase 1.
- Multi-cloud patterns may be added later as examples, not as initial
  service commitments.
- Every scoped implementation pack includes basic handover deliverables.
- Day-2 Operations Handover Pack is for existing systems that need operational
  ownership transfer, not for charging separately for basic project handover.
- Monthly Technical Advisory is a future continuous service and requires a
  separate boundary for hours, response expectations, and production risk.

## Offer Statements

- Cost hygiene review with prioritized cleanup candidates, so teams can find
  obvious spend issues before deeper FinOps work starts.
- Architecture review with reliability risk mapping, so teams can decide what
  to fix first instead of collecting generic best practices.
- Day-2 operations handover for existing systems, so a designated engineer can
  follow documented checks, triage paths, escalation triggers, and routine
  operation steps without undocumented verbal dependency.
- Terraform foundation setup for `dev` / `stg` / `prd`, so teams can
  provision environments consistently without ad-hoc manual setup.
- CI/CD bootstrap with safe deploy flow, so deployments are repeatable,
  reviewable, and easier to roll back.

## Not Primary Offer (for now)

- Full observability overhaul.
- SLO and error budget program design.
- Incident response retainer or 24x7 on-call as the core product.
- Multi-cloud implementation from day one.
- Monthly advisory without a separately agreed support and responsibility
  boundary.

## Private and Public Policy

### Default Non-Public Assets

- Working notes and internal drafts.
- Client-derived artifacts and review comments.
- Environment-specific commands.
- Billing, IAM, org inventory, and internal process details.

### Public-Eligible Assets (after sanitization)

- General service definitions.
- Reusable templates.
- Generalized case studies.
- Public-safe runbook examples.
- Reference architecture examples without client identifiers.

### Sanitization Rules

- Remove client names, project IDs, cost numbers, and private endpoints.
- Replace organization-specific values with placeholders.
- Keep only reusable patterns and neutral examples.
- When in doubt, generalize the pattern first before publishing.

## Why This Positioning

- It matches existing experience and current strengths.
- It creates tangible outputs clients can verify quickly.
- It avoids overpromising in weak-fit domains.
